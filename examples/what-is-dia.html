what it is · 02

The saved file is plain HTML

A deck is a closed profile of HTML — a small grammar of roles, tokens, and attributes that a validator can hold forever.

  • Open it anywhere; it presents itself. No app required to read it.
  • Every style reads a token, so one edit restyles the whole deck.
  • Any agent or editor can keep working on it — there is no private format, no build step, no lock-in.

The grammar in one breath:

section.dia-slide per slide · text roles dia-kicker / dia-title / dia-body / dia-caption · layout dia-columns / dia-stack / dia-figure · diagrams as svg.dia-scene · behavior as data-dia-* attributes · the unconvertible preserved verbatim in islands.

what it does · 03

One file, four surfaces

  • 1Edit — a WYSIWYG editor: click text to type, drag elements and diagram nodes, restyle by role and token, undo everything.
  • 2Present — the deck runs itself in any browser: arrows navigate, data-dia-step reveals in order.
  • 3Import — foreign decks (reveal, Marp, bespoke JS) convert through a reviewed, measured pipeline.
  • 4Agents — a copilot that proposes changes you preview and accept; skills, an AGENTS.md manual, and an MCP server for every coding tool.
how it works · import · 04

Foreign decks convert through a measured pipeline

The original executes in a sandbox — its own scripts run, fonts load, every slide is sampled in its activated state.

Conversion carries what was measured: positions become spacing, colors and faces bind to tokens, animations become build steps, custom bullets become marker tokens.

The review proves fidelity per slide with a visual-consistency score — and anything the dialect can't express is preserved verbatim, never approximated.

how it works · verification · 05

Conversion is verified, not trusted

Every imported slide is rendered side by side with its original and scored on four named axes:

  • Ddisplacement — how far content moved (graded, not a cliff)
  • Llayout — does the composition hold
  • Aappearance — colors, typography, surfaces
  • Ccompleteness — dropping content can never win the score

The score is a ratchet: golden decks live in a corpus with pinned per-slide baselines, and any change that lowers one fails the test suite until a human decides otherwise.

Repairs are model-assisted but measured — a repair that scores lower, or deletes source text, is rejected automatically. Every candidate stays selectable; the human outranks the metric.

the original of every imported slide ships inside the deck — comparison works forever, no source file required

how it works · diagrams · 06

Scenes: semantics in, geometry out

A diagram is an svg.dia-scene: nodes carry data-x/y/w/h and a shape; edges name their endpoints. You author intent; the runtime computes geometry — move a node and its edges reroute around obstacles.

Twelve shapes plus freeform path outlines with rotation; point-edit any path; draw freehand; promote imported art into nodes. What the vocabulary can't hold stays verbatim and stays editable.

how to use it · 07

For hands and for agents

From a terminal:

  1. dia new talk.html — a profile-valid starting deck
  2. dia talk.html — the editor, ⌘S writes back
  3. dia ingest foreign.html — convert through review
  4. dia validate talk.html — the contract gate
  5. dia present talk.html — it runs itself

From a coding agent:

  • Claude Code: /plugin marketplace add pedapudi/diastil
  • Codex, opencode, Antigravity, Cursor: dia agents-md >> AGENTS.md
  • no shell? dia mcp — the same operations as MCP tools
  • the validator is the real interface: an agent that satisfies dia validate has produced a working deck

the invariants · 08

Plain files. Verified conversion. No lock-in.

The editor is complete without a model. The saved deck is plain HTML. Conversion is proven with pixels, never assumed. And nothing in the document path imports anything — a deck you make today opens in twenty years.
